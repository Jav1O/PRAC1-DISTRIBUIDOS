SISTEMAS DISTRIBUIDOS
Prac2

1. Objetivo
Se implementa el servicio de tuplas <key, value1, value2, value3> con una arquitectura cliente-servidor basada en sockets TCP.

2. Componentes reutilizados
- claves.h: se mantiene la API original sin cambios.
- claves.c: se reutiliza la lógica del servicio y su protección mediante mutex.
- app-cliente.c y clientes auxiliares: se reutilizan sin cambios de código fuente.

3. Arquitectura
- El cliente enlaza con libproxyclaves.so.
- proxy-sock.c implementa la API del lado cliente y traduce cada llamada a una petición TCP.
- servidor-sock.c escucha en un puerto TCP, acepta conexiones concurrentes y delega la operación real en libclaves.so.

4. Protocolo de aplicación
El protocolo es binario y explícito. No se envían estructuras de C directamente por el socket. Cada petición contiene:
- código de operación
- longitud de key
- longitud de value1
- dimensión de value2
- enteros x, y, z de value3
- bytes de key
- bytes de value1
- elementos de value2 serializados uno a uno

Cada respuesta contiene:
- status
- longitud de value1
- dimensión de value2
- enteros x, y, z
- bytes de value1
- elementos de value2 serializados uno a uno

Los enteros se codifican en orden de red. Los floats se transmiten como IEEE-754 de 32 bits convertidos a orden de red campo a campo.

5. Configuración y ejecución
Servidor:
- ./servidor 4500

Cliente:
- env IP_TUPLAS=127.0.0.1 PORT_TUPLAS=4500 ./app_cliente_sock

6. Códigos de retorno
- 0: operación correcta.
- -1: error del servicio o validación local de parámetros.
- -2: error de comunicación TCP o imposibilidad de contactar con el servidor.

7. Concurrencia
El servidor es concurrente mediante un hilo desacoplado por conexión. La atomicidad del acceso a las tuplas se mantiene en claves.c mediante un mutex global.

8. Pruebas
La validación se automatiza con run_tests.sh y run_extra_tests.sh, cubriendo:
- funcionalidad básica
- errores del servicio
- error de comunicación
- validación de configuración mediante IP_TUPLAS y PORT_TUPLAS
- resolución por IP y por nombre de host
- límites máximos válidos de key, value1 y value2
- puerto incorrecto y reinicio del servidor
- colisión concurrente sobre la misma clave
- concurrencia
- estrés básico
